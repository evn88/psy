# Архитектура тем и правила стилизации

## 1. Источник истины

В проекте тема строится вокруг трёх слоёв:

1. `src/shared/lib/theme.ts`
   Содержит union-типы темы, ключи cookie/storage, а также helper-функции для SSR и `generateViewport`.

2. `src/components/theme-provider.tsx`
   Единая клиентская точка интеграции темы. Здесь обёртка над `next-themes`, синхронизация cookie для SSR и локальный
   `useTheme`.

3. `src/app/[locale]/(main)/layout.tsx`
   Серверный layout читает cookie темы, подставляет начальный класс на `html` для явных тем (`light` / `dark`) и
   формирует `viewport` metadata.

Важно:
Во всём приложении запрещено импортировать `useTheme` и `ThemeProvider` напрямую из `next-themes`. Используем только
`@/components/theme-provider`.

## 2. Почему реализация устроена именно так

- `next-themes` нужен для корректного раннего применения темы до первого paint.
- Cookie нужна серверу, чтобы SSR и `generateViewport` знали предпочтение пользователя.
- `localStorage` и реакция на `prefers-color-scheme` остаются на стороне `next-themes`.

Итог:
Пользователь видит правильную тему сразу, без вспышки светлой темы при `theme=system`.

## 3. Где хранить токены темы

Все глобальные цветовые и семантические токены темы должны жить только в `src/app/globals.css`.

Правильная структура:

- `:root` содержит светлые значения по умолчанию.
- `.dark` содержит только переопределения для тёмной темы.

Что считается токеном:

- `--background`, `--foreground`, `--border`, `--muted`
- брендовые переменные вроде `--color-pink`
- общие токены лендинга вроде `--landing-bg`, `--landing-text`

Нельзя:

- дублировать одни и те же цвета по разным CSS Modules
- заводить новые глобальные токены внутри компонентных CSS-файлов
- копировать hex/hsl-значения в Tailwind-классах, если уже есть семантический токен

## 4. Где писать стили

Используем следующий порядок:

1. Tailwind utility-классы для локальной раскладки, отступов, размеров и простых состояний.
2. Глобальные CSS variables из `globals.css` для цветов, фонов, бордеров и общих theme-aware значений.
3. CSS Modules только для изолированной компонентной композиции, сложной анимации, декоративных слоёв и случаев, где
   utility-классы начинают дублироваться.

Правило:
Если компоненту нужен theme-aware цвет, сначала ищем существующий токен в `globals.css`, а не создаём новую пару
`light/dark` прямо в модуле.

## 5. Как правильно делать theme-aware UI

### Если можно решить через CSS

Предпочтительный вариант:

- использовать семантические utility-классы (`bg-background`, `text-foreground`, `border-border`)
- использовать `dark:` только для локальных исключений, когда семантического токена недостаточно

### Если нужно поведение, зависящее от темы

Например:

- переключить иконку солнце/луна
- выбрать картинку
- изменить runtime-поведение стороннего редактора

Тогда используем `useTheme()` из `@/components/theme-provider`.

Разница между значениями:

- `theme` — выбор пользователя: `light | dark | system`
- `resolvedTheme` — фактическая тема интерфейса: `light | dark`

Правило:
Для UI-логики, которая должна понимать реальный результат системной темы, используем `resolvedTheme`.

## 6. Гидратация и anti-patterns

`theme` и `resolvedTheme` нельзя считать надёжно известными на сервере для ветвления JSX.

Если компонент рисует разный UI в зависимости от темы:

- рендерим нейтральный placeholder до mount
- либо скрываем theme-зависимый UI до клиента

Подходы, которые в проекте допустимы:

- `useSyncExternalStore` для флага mounted
- нейтральная иконка/плейсхолдер до гидратации
- CSS-скрытие двух версий UI, если это выгоднее и без лишнего дублирования

Нельзя:

- рендерить theme-sensitive UI на сервере, если он зависит от `resolvedTheme`
- вручную читать `window.matchMedia` в каждом компоненте
- вручную писать в `localStorage` или cookie из произвольных мест приложения

Единственное место, где cookie темы синхронизируется на клиенте, это `src/components/theme-provider.tsx`.

## 7. Как менять тему

Тему меняем только через:

```tsx
const {setTheme} = useTheme();

setTheme('light');
setTheme('dark');
setTheme('system');
```

Нельзя:

- напрямую менять класс `html`
- напрямую писать `document.cookie = "theme=..."`
- напрямую писать в `localStorage['theme']`

Исключение возможно только при интеграции сторонней библиотеки, которая рендерит portal вне дерева компонента и не умеет
читать тему через контекст. Даже в этом случае:

- изменение должно быть локализовано
- должен быть cleanup
- нельзя подменять глобальную тему приложения, только адаптировать сторонний виджет

## 8. Как добавлять новые theme-aware компоненты

Минимальный чеклист:

1. Проверить, можно ли обойтись одними семантическими токенами и Tailwind-классами.
2. Если нужен runtime-доступ к теме, сделать компонент клиентским.
3. Использовать `useTheme()` только внутри этого клиентского компонента.
4. Не рендерить зависящие от темы детали до mount без placeholder.
5. Не дублировать уже существующие токены и цвета.

## 9. Как добавлять новые токены

Перед добавлением нового токена:

1. Проверить, не покрывает ли задачу существующий `--background`, `--card`, `--muted`, `--foreground`, `--primary` и
   т.д.
2. Если токен реально новый, добавить его сразу в обе темы:
   светлая версия в `:root`, тёмная в `.dark`.
3. Дать токену семантическое имя, а не визуальное.

Хорошо:

- `--calendar-slot-bg`
- `--marketing-divider`
- `--editor-toolbar-border`

Плохо:

- `--blue-2`
- `--dark-card-2`
- `--pink-hover-final`

## 10. Что делать с CSS Modules

В CSS Modules не дублируем полную светлую и тёмную палитру.

Правильно:

```css
.card {
    background: hsl(var(--card));
    color: hsl(var(--card-foreground));
    border-color: hsl(var(--border));
}
```

Допустимо:

```css
:global(.dark) .specialSurface {
    box-shadow: 0 12px 40px rgba(0, 0, 0, 0.35);
}
```

Но только если это действительно локальная декоративная корректировка, а не попытка заново описать тему компонента
вручную.

## 11. Архитектурные запреты

Запрещено:

- прямые импорты из `next-themes` вне `src/components/theme-provider.tsx`
- создание альтернативных провайдеров темы
- хранение theme-токенов в нескольких местах
- ручное дублирование theme logic в хуках страниц и виджетов
- разрастание `.dark ...`-правил по всему проекту, если можно использовать семантические токены

## 12. Быстрый ориентир по файлам

- `src/shared/lib/theme.ts` — типы, ключи, SSR helper-функции
- `src/components/theme-provider.tsx` — клиентский провайдер и `useTheme`
- `src/app/[locale]/(main)/layout.tsx` — серверная интеграция темы в layout
- `src/app/globals.css` — глобальные токены и theme overrides

Если новая задача по теме не укладывается в эту схему, сначала нужно обновить архитектуру, а не добавлять точечный
обходной код.
